# SecureAgent: 具備完整安全生命週期的 AI Agent 控管系統

在將大語言模型 (LLM) 推向生產環境時，最棘手的往往不是如何觸發工具，而是如何確保系統不被惡意使用者玩壞。

本專案實作了一個 **SecureAgent** 機制，將安全防護思維深度整合進 AI Agent 的生命週期中。透過輸入防禦、輸出遮罩、結構化輸出驗證、以及高風險操作的人工介入 (Human-in-the-loop)，建立起一套多層次的 AI 安全護欄 (Guardrails)，有效防止提示詞注入 (Prompt Injection) 與敏感隱私資料 (PII) 的外洩。

---

## 🔒 核心安全防護機制 (Security Guardrails)

本專案將 AI 互動的完整生命週期切分為四道安全防線，確保任何階段的潛在威脅都能被精準攔截：
### 1. 輸入端安全控管 (Input Guardrail)
* **原理**：在請求送出給大模型之前，先利用正則表達式 (Regex) 針對常見的攻擊樣式進行黑名單過濾。
* **防禦對象**：
  * **提示詞注入 (Prompt Injection)**：攔截如 `ignore previous instructions` 或 `system prompt` 等意圖劫持模型行為的指令。
  * **系統命令注入 (Command Injection)**：攔截惡意的系統層級指令（如 `rm -rf` 或 `format c:`）。

### 2. 輸出端結構化驗證 (Structured Output Validation)
* **原理**：採用 PyTorch/Pydantic 的強型別資料結構定義 `AgentResponse`，並結合 OpenAI 的 `beta.chat.completions.parse` 進行強行約束。
* **好處**：確保 LLM 輸出的 JSON 欄位完全符合預期。即使模型受到干擾，也無法透過輸出混亂的非結構化文字來繞過系統邏輯，並能預先捕捉 `ValidationError` 進行熔斷處理。

### 3. 工具執行安全控管 (Human-in-the-loop)
* **原理**：實作**風險分級控管**。當 Agent 研判需要觸發外部工具時，系統會自動比對高風險工具清單。
* **機制**：
  * **低風險操作**：自動流暢執行並記錄日誌。
  * **高風險操作**（如：刪除資料庫、發送發票、執行系統命令）：觸發中斷機制，在終端機彈出提示，必須由管理員手動輸入 `yes` 授權後方可執行，杜絕 AI 擅自做出破壞性行為。

### 4. 敏感資料過濾與遮罩 (PII Masking)
* **原理**：在最終回覆呈現給使用者前的最後一刻，利用安全審查機制（Scrubbing）掃描內文。
* **效果**：自動識別並遮罩機密資訊（例如：電子郵件、OpenAI `sk-` 開頭的 API 密鑰樣式），將其替換為 `[REDACTED_API_KEY]` 或 `[REDACTED_EMAIL]`，避免模型在推理過程中意外將後台機密「吐」給前端使用者。

## 🛠️ 模組超參數與環境配置

* **基礎核心**：基於 `gpt-4o` 模型進行高精確度的意圖解析與結構化輸出。
* **安全日誌**：內建 `logging` 追蹤機制，所有安全警報（如攔截注入攻擊、遮罩敏感資料、使用者拒絕授權）皆會同步輸出至控制台，並永久保存至本地的 `agent_security.log` 檔案中，便於後續的資安審計（Audit Trail）。

---

## 🚀 驗證與測試場景

本專案內建三大完整測試情境，用以驗證安全防禦的有效性：

1. **正常安全請求測試**：驗證常規對話與流程順暢度。
2. **提示詞注入攻擊測試**：輸入惡意的逆向工程指令，驗證系統是否能成功熔斷並給出「系統檢測到不安全的請求，拒絕處理」的安全回覆。
3. **隱私資料外洩防禦測試**：模擬 LLM 意外吐出敏感金鑰時，驗證系統能否在毫秒級內完成 `[REDACTED]` 遮罩。
